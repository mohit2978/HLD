# Async communication

At its core, asynchronous communication relies on sharing data between components in a non-blocking way. This data exchange can take many forms, but messaging systems are among the most common.

## What Is a Message?
A message is the fundamental unit of communication in asynchronous systems. It consists of:

- Header: Key-value pairs that describe metadata, such as the sender, timestamp, message type, or priority.

- Body: The actual data payload, which can be in text or binary format. The format of the body is not predefined (e.g., JSON or XML) and must be defined by the system’s creators.

```text
+-----------------------------------------------+
| Header                                        |
| message_id: 12345, timestamp: 1678901234567   |
+-----------------------------------------------+
| Body                                          |
| { "order_id": "A456", "status": "confirmed" } |
+-----------------------------------------------+
```

In this example using RabbitMQ, the message header includes a unique message_id for tracking and a timestamp indicating when the message was created. The body contains an order ID and its current status, demonstrating a common use case in e-commerce systems to asynchronously notify inventory services about new orders.

## How Asynchronous Messaging Works
Asynchronous messaging systems enable different styles of communication based on the use case:

### One-Way Notification
A sender transmits a message to a recipient without expecting a response. This is useful for fire-and-forget scenarios, such as logging or sending alerts.

### One-Way Pub/Sub
A sender publishes messages to a topic, and multiple subscribers listen for updates. This is common in event-driven architectures where many components react to the same event.

### Two-Way Async Request-Response Style
The sender sends a request message and later receives a response message. This requires two channels. This style is not as widely used as the first two.

## Message Broker
You may have noticed the green “message channel” in the figures. This is typically implemented using a message broker, an intermediary through which all messages flow. A sender writes the message to the message broker, and the message broker delivers it to the receiver. It ensures reliable delivery, manages message queues, and supports various messaging patterns like point-to-point and publish-subscribe.

Popular message brokers include RabbitMQ, Apache Kafka, and Amazon SQS. Each message broker implements the message channel concept in a different way.

```text
| Message Broker | Channel Name Type |
|---------------|--------------------|
| RabbitMQ      | Queues, Topics     |
| Apache Kafka  | Topics (Partitions)|
| Amazon SQS    | Queues             |
| Google Pub/Sub| Topics             |
```

## Pros and Cons of Asynchronous Communication
### Pros
- Loose Coupling: Components don’t need to know about each other’s existence.
- Buffering: Messages can be stored temporarily, allowing receivers to process them at their own pace.
- No Service Discovery Needed: Senders only interact with the message queue or topic, avoiding the complexity of finding specific receivers.
### Cons
- Complexity: Managing message brokers and ensuring reliability can add operational overhead.
- Latency: Asynchronous systems might introduce delays, especially when messages accumulate in queues.
- Error Handling: Debugging can become more challenging due to the decoupled nature of the system.

## Message Queues

A message queue (also called message broker) is a software component that allows communication between different applications or subsystems of an application by passing messages. An analogy is that it’s like emails, not between people but rather between app services.

The messages are usually small, and can be things like requests, replies, error messages, or just plain information. To send a message, a component called a producer adds a message to the queue. To receive a message, a component called a consumer retrieves the message, processes the message, and acknowledges that the work is done by removing the message from the queue. The queue is a buffer between producers and consumers.

### Main advantages of using message queues:

1. Acts as a buffer between services when the producer and consumer have different processing rates.

    Message queues can help manage the difference in processing rates between the producer and the consumer. For example, services that involve writing to a database are generally slower than stateless web services that process requests in memory. By using a message queue, the producer can send messages at its own pace without overwhelming the slower consumer.

Example:

    - In a photo-sharing application, users may upload photos rapidly, and the producer sends the photos to the message queue quickly. However, the photo uploader and database updater services might not be able to process the images at the same speed. Using a message queue allows the consumer services to process the images at their own pace without losing any data.

    - On a video-sharing platform like YouTube, users may post comments at a high rate. The front-end service can handle posting comments quickly, but the database updater service responsible for storing the comments might not keep up with the same speed. Using a message queue allows the database updater to process the comments at its own pace without losing any data.
2. Decouples components (producer and consumer). If the consumer goes down, the message is still in the queue and can be processed when it comes back up.

    - Message queues provide a level of fault tolerance by decoupling the producer and consumer components. If a consumer service goes down or experiences an issue, the messages remain in the queue, ready to be processed when the consumer comes back online.

    - In the previous examples, if either photo uploader or database updater goes down, the message is still in the queue and can be processed when the service recovers.


## Two types of message queues

### AMQP/JMS-style message queue
These message brokers are based on the Advanced Message Queuing Protocol (AMQP) or Java Message Service (JMS) standards. These are originally designed to pass messages in large enterprise systems. They are designed to support various messaging patterns like publish/subscribe, point-to-point, and request/reply.

- Messages are typically held in memory, and once they are consumed by the intended recipient, they are deleted. These brokers emphasize message routing, filtering, and transformation.
- There could be duplicate messages and ordering is either not guaranteed or per-queue guaranteed (no global ordering).
Examples: RabbitMQ, ActiveMQ, Azure Service Bus, IBM MQ

### Log-based message queue
These message queues, such as Kafka or AWS Kinesis, use a distributed, append-only log for storing and processing messages. The broker assigns all messages in a partition to the same consumer node, and always delivers messages in the same order. Parallelism is achieved through partitioning, and consumers track their progress by checkpointing the offset of the last message they have processed. The broker retains messages on disk, so it is possible to jump back and reread old messages if necessary.

Note that log-based message queues are often referred to as message streaming platforms. Kafka’s official definition is “an open-source distributed event streaming platform”. The main difference between a stream and a queue is that messages sent to a stream are persisted and can be read back using an iterator, while messages sent to a queue are processed independently and deleted when done. We will discuss more about stream processing in a later chapter.

- Messages are persisted on disk, allowing for long-term storage, reprocessing, and increased fault tolerance.
- They focus on high-throughput, low-latency processing, and are optimized for data streaming and real-time analytics.
- Log-based brokers often provide strong durability guarantees and can handle high volumes of data.
- Messages within partitions are totally ordered, and each message has a unique offset for maintaining the order.
Examples: Apache Kafka, AWS Kinesis, Google Cloud Pub/Sub, Azure Event Hubs

| Feature | AMQP/JMS-style Traditional Message Queue | Log-based Message Queue |
| :--- | :--- | :--- |
| **Message Storage** | In-memory (usually transient) | Persistent (on-disk storage) |
| **Message Durability** | Optional (configurable) | Strong durability guarantees |
| **Message Ordering** | Unordered or per-queue ordering | Partition-based ordering |
| **When to use** | Messages are expensive to process, order not important | Messages are fast to process, ordering is important |
| **Example use case** | Encoding videos, fetching hotel prices, downloading large files | Analytics stream processing, event sourcing (recording video game moves, payment system processing), IOT sensor data |
