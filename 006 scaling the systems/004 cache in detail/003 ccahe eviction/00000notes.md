# Cache eviction

After choosing a caching pattern, you must decide what happens when cache fills up. Cache lives in memory. Memory is limited and expensive. When cache reaches capacity and new data arrives, the system must evict existing items to make room.

Cache eviction policies determine which items to remove when this happens. The choice affects hit rates, latency, and system performance. Different applications need different policies based on their access patterns.

## Least Recently Used

LRU evicts the item that has not been accessed for the longest period. This assumes items not accessed recently will not be accessed soon. A web browser cache uses LRU. When you visit webpages, the browser stores images, scripts, and stylesheets. When cache fills, the browser removes items from pages you have not visited recently


LRU is widely used because of its simplicity and effectiveness. It performs poorly when frequently accessed items are followed by long periods of inactivity. The long gap causes the item to be evicted even though it will be needed again soon.


## Most Recently Used
MRU evicts the most recently used items first. This is the opposite of LRU. It works when newer items are less likely to be accessed again soon. A music streaming app might use MRU. Songs in the listening queue stay in cache. The song that just finished playing gets evicted first when cache needs space. The user already heard it and will likely skip to a different song.

MRU is used less often than LRU but fits specific scenarios where recent access indicates the item is finished rather than hot.

## Least Frequently Used
LFU evicts the item accessed least often. It counts accesses and removes items with the lowest count. This works when access patterns are bursty but predictable. An e-commerce app might use LFU for product recommendations. Popular products viewed frequently stay in cache for quick loading. Unpopular items get evicted first.

The downside is that LFU keeps older frequently used data if a burst of different data is accessed temporarily. An item accessed many times in the past but not recently can stay in cache while newer items get evicted.

## Time To Live
TTL evicts data after a duration. The duration is set when data enters the cache. This works when data becomes stale or irrelevant after a period. A weather app caches forecasts with TTL. Weather changes regularly. When the TTL expires, the app removes the forecast and fetches updated data. The cache automatically stays fresh without manual invalidation.

TTL is essential for time-sensitive data like weather forecasts, stock prices, or session tokens.


### Why do we need TTL
Time to live is used to get rid of data that might stay around for too long, ensuring that the cache remains up-to-date and efficient.

### How to decide what TTL to use
Redis does not have a default time to live, but it can be assigned during creation and modification or after the fact.

It's important to set the time to live (TTL) based on what you're doing. Redis does not have a default TTL. Objects are by default permanent. You can set it for a fixed time or have it expire after a certain amount of time. It's essential to consider the nature of the data and how frequently it is accessed or updated. Choose a TTL that ensures data remains relevant without causing unnecessary evictions and cache misses. Here are some examples of picking TTL:

### TTL Configuration by Use Case

| Use Case | TTL Value | Explanation |
| :--- | :--- | :--- |
| **User session data** | 1800s (30 minutes) | Matches standard session timeout, removing data once the user is inactive. |
| **API rate limiting** | 3600s (1 hour) | Matches the rate limiting window, resetting the request counter automatically. |
| **Cached search results** | 300s (5m) - 86400s (24h) | Varies based on data update frequency and acceptable staleness for the user. |
| **Configuration data** | None or 2592000s (30 days) | Rarely changing data; stored until explicitly updated or a long-term cache flush. |
| **Weather data** | 600s (10 minutes) | Matches meteorological update cycles, ensuring forecasts remain accurate. |

---

### Why TTL Selection Matters

In System Design, choosing the correct TTL is a balance between **Database Load** and **Data Accuracy**.



* **Short TTL:** Higher database load (more cache misses) but very fresh data.
* **Long TTL:** Lower database load but higher risk of serving "stale" or outdated information to users.

#### 1. The "Sliding Window" Pattern
For **User Sessions**, we often use a "Sliding TTL." Every time the user makes a request, the 30-minute timer is reset to zero. This ensures the session only expires after 30 minutes of *inactivity*, not 30 minutes after login.



#### 2. The "Stale-While-Revalidate" Pattern
For **Weather Data** or **Search Results**, some systems serve the "stale" (expired) data to the user immediately while simultaneously triggering a background process to update the cache. This ensures the user never experiences the latency of a "Cache Miss."

## Random Replacement
Random Replacement evicts random items from cache. It is the simplest policy. It ignores access patterns and frequency. This makes it less effective than other policies. It might be useful when other policies add too much overhead or when access patterns are truly random.

A news app with continuously updated articles might use Random Replacement. Articles age quickly and are replaced by newer ones. Since no article stays relevant long, randomly replacing cached articles works as well as tracking access patterns.

## Choosing the Right Policy

### Cache Eviction Policies: How Systems Free Up Space

When a cache becomes full, an **Eviction Policy** (or Cache Replacement Algorithm) determines which existing data must be removed to make room for new data.

| Policy | Description | Example |
| :--- | :--- | :--- |
| **Least Recently Used (LRU)** | Evicts the data that has not been accessed for the longest time. | **Web browser cache:** Removing data from webpages you haven't visited in a while. |
| **Most Recently Used (MRU)** | Evicts the most recently accessed data. | **Music streaming app:** Discarding the song just played from the cache to make room for the next. |
| **Least Frequently Used (LFU)** | Evicts the data that has the lowest access count (frequency). | **E-commerce app:** Removing less popular product pages that rarely get hits. |
| **Time To Live (TTL)** | Evicts data automatically after a specific lifespan has passed. | **Weather app:** Removing cached forecasts after 30 minutes because they are "expired." |
| **Random Replacement (RR)** | Evicts random data when the cache is full without tracking history. | **News app:** Randomly replacing cached articles when the device memory is low. |

---

### Visualizing the Algorithms



#### 1. LRU (Least Recently Used)
This is the **most common** policy in System Design. It assumes that if you haven't used data in a long time, you probably won't use it soon. It is usually implemented using a **Doubly Linked List** and a **Hash Map** to achieve $O(1)$ time complexity for both access and eviction.



#### 2. LFU (Least Frequently Used)
LFU is smarter than LRU in cases where some items are "always popular" but haven't been touched in the last few minutes. However, it is more complex to implement because it requires maintaining a "hit counter" for every single item in the cache.

#### 3. TTL (Time to Live)
TTL is often used alongside other policies. Even if a piece of data is "Recently Used," it might be legally or logically required to be deleted after a certain time (e.g., a session token that expires after 2 hours for security).



---

### Interview Summary: Which one to pick?
* **Pick LRU** for general-purpose caching where "recency" is a good predictor of future use.
* **Pick LFU** if your data has "hot" items that stay popular over very long periods.
* **Pick TTL** when your data has a strict "expiration date" (like stock prices or weather).
The correct policy depends on application nature, access patterns, and performance requirements. For predictable usage where users revisit the same data within short intervals, LRU works best. Session data and browser caches fit this pattern. Users access the same resources repeatedly during a session.

For systems where some data is accessed far more frequently than others, LFU is ideal. Product recommendations and trending content see clear frequency differences. The most popular items should stay cached.

When new data becomes obsolete quickly, consider MRU. Playlist queues in streaming apps fit this pattern. Recently played items are done and can be evicted. Upcoming items need to stay ready.

For unpredictable access patterns where tracking usage is not feasible, Random Replacement might suffice. When data changes so frequently that caching is not critical, the overhead of tracking patterns outweighs the benefit.

TTL is essential when cached data is time-sensitive. Weather forecasts, financial data, and promotional prices all expire naturally. TTL ensures data refreshes automatically after the appropriate period.

## Cache Invalidation
Eviction policies alone do not ensure strict consistency between cache and database. Cache invalidation strategies remove or update stale data promptly. Write invalidation removes data from cache when it updates in the primary database. Event-driven invalidation clears specific entries based on triggers like database changes or user actions. Manual invalidation lets developers control when to remove data in critical applications.

### Multi-Level Caching
Large-scale systems often use multiple cache layers with different policies. A local in-memory cache might use LRU for speed. A distributed cache like Redis uses LFU to retain globally popular data. A CDN caches static resources at the edge using TTL for freshness. The application server uses LRU for dynamic content accessed recently.

This layering optimizes each level for its purpose. Edge caches prioritize freshness. Application caches prioritize recent access. Distributed caches prioritize global popularity.

### Tuning for Performance
Set cache size based on workload and available memory. Too small causes frequent evictions and low hit rates. Too large wastes resources without improving performance. Monitor the cache hit ratio, which is cache hits divided by total requests. A low hit ratio indicates the cache is not utilized effectively. You might need a different eviction policy or larger cache.

Tune policy parameters based on behavior. Adjust TTL durations to match data freshness requirements. Balance LFU weights to avoid keeping stale but historically popular data. Test different policies under realistic load to measure hit rates and latency.
